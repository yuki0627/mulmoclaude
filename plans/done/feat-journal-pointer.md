# Feature: expose journal summaries as discoverable context in new sessions

## Goal

Let every new chat session know that the **workspace journal** (auto-generated by PR #107) exists and where to find it, so the LLM can opportunistically read past summaries when the current user question would benefit from prior context.

Critically: without bloating the system prompt, without loading summaries wholesale, and without persisting the pointer into the chat log the user sees.

## Non-goals

- **Not** adding journal content to the system prompt. `memory.md` and the wiki hint already live there; the journal deliberately stays out so the system-prompt doesn't grow unbounded as the journal accretes topics and daily files.
- **Not** eagerly loading summary files. The LLM decides per-question whether to Read anything, using its existing built-in Read / Grep / Glob tools.
- **Not** injecting the pointer into the persisted jsonl chat log. The user's visible chat history stays as the user typed it.
- **Not** every-turn repetition. Once is enough — subsequent turns of the same session resume via `--resume <claudeSessionId>` and inherit the pointer from the first turn's context.
- **Not** per-role opt-out in v1. Keep the logic uniform; add role-level granularity later if real need surfaces.

## Design

### Where the pointer goes

The pointer is prepended to the **user prompt** of the **first turn** of a session. Specifically:

1. `server/api/routes/agent.ts` receives `POST /api/agent` with `{ message, chatSessionId, ... }`
2. The original `message` is appended to `workspace/chat/<chatSessionId>.jsonl` unchanged (so the chat log stays clean)
3. `claudeSessionId` is resolved (from in-memory map or `readClaudeSessionId`)
4. If `claudeSessionId` is **absent** (first turn of this session), a helper decorates the message with the pointer
5. The decorated message is passed to `runAgent`, which ultimately reaches `claude -p <decoratedMessage>`

```ts
// server/api/routes/agent.ts sketch
const claudeSessionId =
  claudeSessionMap.get(chatSessionId) ??
  (await readClaudeSessionId(metaFilePath, resultsFilePath));

// Decorate only on the first turn — subsequent turns resume via
// --resume, so the pointer is already in Claude's context from turn 1.
const decoratedMessage = claudeSessionId
  ? message
  : prependJournalPointer(message, workspacePath);

for await (const event of runAgent(
  decoratedMessage,
  role,
  workspacePath,
  sessionId,
  PORT,
  claudeSessionId,
  pluginPrompts,
  systemPrompt,
)) {
  // ...unchanged
}
```

### Helper: `prependJournalPointer(message, workspacePath)`

Pure function living in `server/agent/prompt.ts` alongside the existing `buildMemoryContext` and `buildWikiContext` — same style, same file, so tests can reach it easily.

```ts
// server/agent/prompt.ts
export function prependJournalPointer(
  message: string,
  workspacePath: string,
): string {
  const indexPath = join(workspacePath, "summaries", "_index.md");
  if (!existsSync(indexPath)) return message;

  const pointer = [
    "<journal-context>",
    "This workspace maintains an auto-generated journal of past",
    "sessions under `summaries/`:",
    "- `summaries/_index.md` — browseable index of topics and recent days",
    "- `summaries/topics/<slug>.md` — long-running topic notes",
    "- `summaries/daily/YYYY/MM/DD.md` — per-day summaries",
    "",
    "If the user's question may benefit from prior context, read",
    "`summaries/_index.md` first with the Read tool, then drill into",
    "relevant topic or daily files. Skip this when the question is",
    "self-contained.",
    "</journal-context>",
    "",
    message,
  ].join("\n");
  return pointer;
}
```

### Pointer content rationale

- **`<journal-context>` XML-ish tags** — matches the `<reference type="memory">` convention used in `buildMemoryContext`. Visually separates the ambient context from the user's actual message.
- **Path listing** — enumerates the three paths Claude needs: `_index.md`, `topics/*.md`, `daily/YYYY/MM/DD.md`. `Read` needs exact paths.
- **"Read `_index.md` first"** — sequences the lookup. Index is small and fast; topic/daily drill-down is on-demand.
- **"Skip this when the question is self-contained"** — opt-out permission. Avoids unnecessary Reads on simple questions.
- **First-turn-only placement** — see the "Why first turn only" section below.

### Why first turn only

Claude Code CLI uses `--resume <sessionId>` for follow-up turns. The earlier turns (including our injected pointer) are already in Claude's context when it resumes. Re-sending the pointer on every turn would:

- Waste ~150 tokens per follow-up
- Confuse Claude ("why are you telling me this again?")
- Bloat the chat-log-as-seen-by-the-model with repeated ambient text

First-turn-only keeps the pointer at the top of the session's Claude context and lets it live there for the session's lifetime (subject to context trimming).

Counter-concern: what if journal content changes mid-session? A new daily summary gets written while the user is chatting. Does Claude miss it?

Answer: the pointer is a **path**, not data. The `Read` tool always returns the **current file content** when invoked. So even an old pointer reference still works for freshly-written files. No staleness.

### Safe default: no pointer if journal doesn't exist

`prependJournalPointer` returns the original message unchanged if `summaries/_index.md` doesn't exist. In workspaces where auto-journal has never run — or has been disabled — nothing is injected. No behavioural regression.

## Out of scope

- **Role-based opt-out**: `minimum` or `free_chat` style roles might want to skip the pointer. Not needed for v1 because the LLM can simply ignore the pointer on simple questions. Add a `Role.skipJournalPointer?: boolean` field later if usage data shows waste.
- **Searching summaries on the LLM's behalf**: a hypothetical `searchJournal(query)` MCP tool. Would be complementary but is a separate feature. The pointer approach requires no new tool; the built-in `Read` / `Grep` / `Glob` handle lookup.
- **Dynamic updates**: pushing "new summary written" notifications to an in-flight session. Out of scope; handled organically on the next session's first turn.
- **Privacy / redaction**: if a past session contained sensitive content that made it into a topic summary, the pointer will point the LLM at it. This is aligned with journal's existing behaviour and is covered (or not) separately by the still-open "delete session" discussion.

## Implementation plan

1. **Add `prependJournalPointer` to `server/agent/prompt.ts`**
   - Pure function, ~20 lines
   - Same file as `buildMemoryContext` / `buildWikiContext` for style consistency

2. **Wire it into `server/api/routes/agent.ts`**
   - Import the helper
   - Resolve `claudeSessionId` (already done)
   - Branch: decorate on first turn, pass-through on resume
   - Pass `decoratedMessage` to `runAgent` instead of `message`
   - **Important**: ensure the `appendFile` to jsonl uses the ORIGINAL `message`, not the decorated one. The jsonl is the user-facing chat log and must stay clean.

3. **Unit test `test/agent/test_agent_prompt.ts`**
   - `prependJournalPointer` with `_index.md` present → contains the pointer + original message at the end
   - `prependJournalPointer` with `_index.md` absent → returns the original message unchanged
   - `prependJournalPointer` with empty-string message → pointer wraps empty string, still valid
   - Pointer contains all three path types (`_index.md`, `topics/`, `daily/`)

4. **Format / lint / typecheck / build / test**

5. **Manual smoke** (post-merge, but noted):
   - Fresh session + non-empty journal → Claude mentions reading `_index.md` if the question is relevant
   - Fresh session + empty workspace (no journal) → no pointer injected
   - Resumed session → pointer not re-injected (verifiable by logging `decoratedMessage` briefly)
   - Original user message preserved in jsonl (grep for the decorated wrapper string, should return zero hits)

## Test plan (automated only)

```text
test/agent/test_agent_prompt.ts — add describe block for prependJournalPointer

  - prepends journal-context block when summaries/_index.md exists
  - preserves original message verbatim at end of decorated output
  - handles empty message
```

Integration smoke tests (server hit, real Claude CLI) are out of scope for the automated suite — the existing journal tests already mock the CLI via `summarize` injection, and this feature doesn't add a new LLM call.

## Risks & mitigations

- **LLM ignores the pointer and never reads summaries** → acceptable, the point is opt-in discoverability not forced ingestion. If this becomes a problem, tighten the wording ("always read _index.md on the first turn") or promote the pointer to a system-prompt section.
- **Pointer triggers unnecessary Read tool calls** → acceptable cost (maybe 1 extra tool call on turn 1 when the LLM investigates). The `<journal-context>` wording explicitly says "skip when self-contained" so simple questions should bypass.
- **Claude's context gets trimmed and the pointer falls out** → rare in practice (first turn is last to be trimmed), but if it happens the follow-up turn loses journal awareness. Acceptable — the user can start a fresh session to re-establish.
- **Workspace without auto-journal installed** → helper detects missing `_index.md` and no-ops. Zero regression.
